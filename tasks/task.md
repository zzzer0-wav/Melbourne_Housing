# Проєкт: прогноз цін на житло в Мельбурні

Навчальний проєкт від початку до кінця: ви берете реальний датасет, очищаєте його, досліджуєте, створюєте нові ознаки і будуєте модель, яка прогнозує ціну житла. Працюєте лише у двох інструментах: **Google Colab** (середовище для коду) і **GitHub** (місце зберігання даних та ноутбука).

Це завдання дає підказки і напрямки - готового коду тут немає навмисно. Ваше завдання - думати над кожним кроком самостійно.

---

## Мета

Відпрацювати повний цикл роботи аналітика даних:
очищення -> дослідження (EDA) -> створення ознак -> модель -> прогноз -> висновки.

Тип задачі: **регресія** (ціль - число, ціна).

---

## Датасет

- Назва: **Melbourne Housing Snapshot**
- Джерело (Kaggle): `https://www.kaggle.com/datasets/dansbecker/melbourne-housing-snapshot`
- Файл: `melb_data.csv`
- Ціль для прогнозу: стовпець `Price` (ціна продажу).

Короткий зміст стовпців: характеристики об'єкта (кількість кімнат `Rooms`, тип житла `Type`, відстань до центру `Distance`, площа ділянки `Landsize`, рік побудови `YearBuilt`, координати тощо), а також район (`Suburb`, `Regionname`, `CouncilArea`), спосіб і дата продажу (`Method`, `Date`).

## Частина 1. Організація роботи (Colab + GitHub)

### 1.1. Створіть репозиторій

1. На GitHub натисніть **New repository**, дайте назву (наприклад, `melbourne-housing`), зробіть його публічним.
2. Додайте файл `README.md` (галочка "Add a README file" при створенні).

Рекомендована структура репозиторію:

```
melbourne-housing/
  data/            # сюди покладете датасет
    melb_data.csv
  notebooks/       # сюди збережете ноутбук з Colab
    solution.ipynb
  README.md        # опис проєкту, кроки, висновки
```

Папку всередині GitHub можна "створити", вказавши шлях у назві файлу при завантаженні (наприклад, `data/melb_data.csv`).

### 1.2. Завантажте датасет з Kaggle

1. Відкрийте сторінку датасета (посилання вище). Для скачування потрібен безкоштовний акаунт Kaggle.
2. Натисніть кнопку **Download** - отримаєте архів. Розпакуйте його і знайдіть `melb_data.csv`.

### 1.3. Додайте датасет у репозиторій

1. У репозиторії натисніть **Add file -> Upload files**.
2. Перетягніть `melb_data.csv`. Щоб він потрапив у папку `data/`, у полі імені вкажіть шлях `data/melb_data.csv` (або спершу створіть папку).
3. Натисніть **Commit changes**.

Зверніть увагу: GitHub приймає файли до 25 МБ через веб-завантаження; наш файл значно менший, тож проблем не буде.

### 1.4. Отримайте RAW-посилання

Щоб Colab міг прочитати файл напряму, потрібне **RAW-посилання** (на "сирий" вміст, а не на веб-сторінку GitHub):

1. Відкрийте `data/melb_data.csv` у репозиторії.
2. Натисніть кнопку **Raw**.
3. Скопіюйте URL з адресного рядка. Він має вигляд:
   `https://raw.githubusercontent.com/<ваш_логін>/<ваш_репозиторій>/main/data/melb_data.csv`

Чому саме RAW: звичайне посилання на GitHub веде на HTML-сторінку з оформленням, а RAW віддає чистий вміст файлу, який бібліотека читання CSV зрозуміє.

### 1.5. Робота в Colab і збереження назад у GitHub

1. Відкрийте `https://colab.research.google.com` -> новий ноутбук.
2. Дані підвантажуйте у пам'ять напряму з RAW-посилання (передайте URL у функцію читання CSV з pandas). Завантажувати файл у Colab вручну не потрібно.
3. Коли закінчите: **File -> Save a copy in GitHub**, оберіть свій репозиторій і папку `notebooks/`. Так ноутбук збережеться у проєкті.

## Частина 2. Завдання покроково

Нижче - кроки і на що звертати увагу. Код пишете самі.

### Крок 1. Завантаження даних

- Підвантажте CSV з RAW-посилання у DataFrame.
- Переконайтеся, що таблиця прочиталася (кількість рядків і стовпців виглядає правдоподібно).

### Крок 2. Перший огляд

Подивіться на дані з кількох боків і занотуйте проблеми:

- розмір таблиці, перші рядки, типи стовпців;
- описова статистика числових стовпців;
- кількість і відсоток пропусків у кожному стовпці.

На що звернути увагу:
- чи немає технічного стовпця-індексу, який потрапив у файл;
- які стовпці мають дуже багато пропусків;
- чи правильно визначилися типи (наприклад, дата може зчитатися як текст);
- чи немає у статистиці підозрілих мінімумів або максимумів.

### Крок 3. Очищення

Це ключовий крок. Перевірте і за потреби виправте:

- **технічний стовпець-індекс** - якщо є, приберіть;
- **дата** - перетворіть текст на справжній тип дати; зверніть увагу на порядок день/місяць;
- **нулі, що насправді означають "немає даних"** - подумайте, чи можлива площа, що дорівнює 0; такі нулі краще вважати пропусками;
- **фізично неможливі значення** - перевірте діапазон року побудови (значення на кшталт 1196 - явна помилка);
- **дублікати** рядків;
- **стратегія щодо пропусків** - для стовпців з дуже високою часткою пропусків подумайте, чи варто їх взагалі використовувати; для решти сплануйте заповнення (числові - якоюсь типовою величиною, категорійні - найчастішим значенням). Важливо: заповнення варто робити так, щоб інформація з тестової вибірки не "просочувалася" у навчання.

### Крок 4. Дослідження (EDA)

Мета - зрозуміти дані перед моделюванням. Подивіться:

- розподіл цілі `Price` (чи він симетричний, чи скошений; що це означає для моделі);
- як ціна залежить від типу житла та району;
- зв'язок числових ознак із ціною (кореляції);
- чи немає пар ознак, які майже дублюють одна одну (підказка: порівняйте `Rooms` і `Bedroom2`).

Робіть прості графіки (гістограма, boxplot, scatter) і формулюйте словами, що бачите.

### Крок 5. Нові ознаки (feature engineering)

Створіть кілька осмислених ознак, наприклад:

- **вік будинку** на момент продажу (з року побудови і дати продажу);
- **частини дати** продажу (рік, місяць);
- **прапорці наявності** для стовпців з пропусками (сам факт пропуску буває інформативним);
- для району з дуже великою кількістю унікальних значень (`Suburb`) подумайте про кодування через частоту, а не через окремий стовпець на кожне значення.

Головне застереження: **не створюйте ознак з цілі**. Будь-що, пораховане з `Price` (наприклад, "ціна за кімнату"), - це витік: у момент реального прогнозу ціни ще немає.

### Крок 6. Підготовка до моделювання

- визначте ціль `y` (`Price`) і ознаки `X` (усе інше осмислене);
- розділіть ознаки на числові і категорійні;
- категорійні закодуйте (для номінальних - One-Hot; зважайте на високу кардинальність);
- зберіть усі кроки підготовки в єдиний конвеєр, щоб застосовувати їх однаково до навчальної і тестової частин;
- поділіть дані на train і test.

### Крок 7. Модель і прогноз

- спершу зробіть **базову лінію** (модель, що завжди передбачає середнє) - щоб було з чим порівнювати;
- навчіть просту модель (лінійна регресія) і складнішу (наприклад, випадковий ліс);
- оцініть на **тестовій** вибірці метриками регресії (середня абсолютна помилка, корінь із середньоквадратичної, коефіцієнт детермінації);
- подивіться на важливість ознак і поясніть словами, що впливає на ціну;
- зробіть прогноз для одного прикладу і порівняйте зі справжньою ціною.

### Крок 8. Висновки і README

- коротко опишіть у README: що за проєкт, які кроки очищення зробили, що показав аналіз, яка модель найкраща і яка її помилка;
- додайте 2-3 ідеї, як покращити результат далі.

## Типові помилки

- оцінювати модель на тих самих даних, на яких вона навчалася (треба на тесті);
- кодувати район числами 0, 1, 2 (це нав'язує неіснуючий порядок);
- заповнювати пропуски, рахуючи статистику на всіх даних разом із тестом (витік);
- створити ознаку з `Price` і отримати фейкову точність (витік);
- залишити текстові стовпці чи пропуски і отримати помилку під час навчання;
- радіти результату, не порівнявши його з базовою лінією.
